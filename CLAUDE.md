# CLAUDE.md — Инструкции разработчика

Этот файл читается Claude Code при каждом запуске.
Следуй всем инструкциям ниже для ЛЮБОЙ задачи в этом проекте.

---

## О проекте

- **Название:** VibeMaster AI
- **Тип:** Telegram чат-бот
- **Стек:** Python 3.12, aiogram, pydantic-settings, SQLite
- **Описание:** Личный ассистент вайбкодера — помогает с генерацией текста и изображений через OpenRouter API

## Цикл разработки — всегда в этом порядке

ПЛАН → АРХИТЕКТУРА → РЕАЛИЗАЦИЯ → ТЕСТЫ → КОММИТ → ДЕПЛОЙ

- Что именно нужно сделать (простыми словами)
- Какие файлы будут созданы или изменены
- Какие внешние сервисы/API задействованы
- Что может пойти не так (граничные случаи)

Показать план и дождаться одобрения перед написанием кода.

## 2. АРХИТЕКТУРА (до реализации)

Спроектируй структуру:
- Точка входа и как запускается
- Разбивка на модули (один модуль = одна ответственность)
- Поток данных: вход → обработка → выход
- Стратегия обработки ошибок

**Правило:** если функция > 30 строк — разбить на части.
**Правило:** если файл > 200 строк — разбить на модули.

## 3. РЕАЛИЗАЦИЯ (писать код)

Строгий порядок:
1. Создать скелет файла/модуля (импорты, подписи функций)
2. Написать основную логику
3. Добавить обработку ошибок
4. Добавить логирование
5. Добавить аннотации типов ко всем функциям
6. Написать docstring для каждой публичной функции

---

## 4. ТЕСТЫ (до коммита)

TDD-цикл для каждой новой функции:
- КРАСНЫЙ  → написать тест (он должен упасть)
- ЗЕЛЁНЫЙ  → написать минимальный код для прохождения теста
- РЕФАКТОР → улучшить код (тесты должны остаться зелёными)

Запускать тесты: `pytest tests/ -v`
Коммитить ТОЛЬКО после прохождения всех тестов.

---

## 5. КОММИТ (после тестов)

```
git add . && git commit -m "тип: описание" && git push
```

Типы коммитов:
- `feat:`     новая функция
- `fix:`      исправление бага
- `refactor:` улучшение кода без изменения поведения
- `test:`     добавление или исправление тестов
- `docs:`     изменение документации

Пушить после КАЖДОГО завершённого фрагмента функциональности.
Это запускает автодеплой на Railway.

## Стандарты кода

### Обязательные правила

**КОММЕНТАРИИ:**
- Все комментарии писать на РУССКОМ языке
- Комментировать каждую нетривиальную строку
- Docstring для каждой функции — на русском

Пример:
```python
def calculate_total(items: list) -> float:
    """Считает сумму всех товаров в корзине."""
    # Суммируем цены всех позиций
    return sum(item.price for item in items)
```

**БЕЗОПАСНОСТЬ — НИКОГДА:**
- Не хардкодить токены, пароли, API-ключи в коде
- Не использовать `print()` вместо `logging`
- Не игнорировать исключения (`except: pass`)
- Не коммитить файл `.env`
- Не создавать функции длиннее 30 строк без разбивки

**ОБРАБОТКА ОШИБОК — ВСЕГДА:**
- `try/except` для всех внешних API-вызовов
- Логировать ошибку через `logger.error()`
- Показывать пользователю понятное сообщение
- Никогда не показывать traceback пользователю

**КОНФИГУРАЦИЯ — ТОЛЬКО ЧЕРЕЗ `.env`:**
- Все секреты через `pydantic-settings` / `os.getenv()`
- Проверять наличие переменных при запуске
- Если переменная не найдена — понятно объяснить что делать

## Безопасность — что отслеживать

Claude Code должен автоматически предупреждать если:

**КРИТИЧЕСКИЕ ПРОБЛЕМЫ (сообщать немедленно):**
- Токен или пароль обнаружен в коде (не в `.env`)
- Отсутствует обработка исключений для внешних вызовов
- SQL-запросы через строковую конкатенацию (SQL-инъекция)
- Пользовательский ввод используется без валидации
- Файл `.env` добавляется в git

**ПРЕДУПРЕЖДЕНИЯ (сообщать в конце задачи):**
- Функция длиннее 30 строк
- Файл длиннее 200 строк
- Дублирование кода (нужно вынести в функцию)
- `print()` вместо `logging`
